# 社区团购管理信息系统前后端服务实现文档

## 一、文档说明

本文档用于指导“社区团购管理信息系统”的前端与后端服务设计和实现。系统采用前后端分离架构，前端提供统一登录入口，并根据用户选择的角色进入不同的业务页面；后端提供统一的用户认证、角色权限控制、商品管理、拼团管理、订单管理、自提核销、支付记录、退款售后和通知管理等服务。

本系统数据库采用精简版 10 张核心表设计：

1. `user` 用户表  
2. `category` 商品分类表  
3. `product` 商品表  
4. `group_activity` 拼团活动表  
5. `group_instance` 拼团实例表  
6. `orders` 订单表  
7. `payment` 支付表  
8. `pickup_point` 自提点表  
9. `refund` 退款售后表  
10. `notice` 通知表

## 二、系统总体架构设计

### 2.1 架构模式

系统采用前后端分离模式：

```text
用户浏览器
   ↓
前端 Vue 项目
   ↓ HTTP / JSON
后端 Spring Boot 项目
   ↓
MySQL 数据库
```

| 层次 | 技术 |
|---|---|
| 前端框架 | Vue 3 |
| 前端路由 | Vue Router |

## 三、角色登录与页面跳转设计

### 3.1 统一登录页面设计

系统前端只设计一个统一登录页面，登录时用户需要填写：

| 输入项 | 说明 |
|---|---|
| 用户名 / 手机号 | 用于识别账号 |
| 密码 | 用于身份验证 |
| 登录角色 | 用户选择本次登录身份 |

登录角色包括：

| 角色值 | 角色名称 | 登录后首页 |
|---|---|---|
| USER | 普通用户 | 购物首页 |
| LEADER | 团长 / 社区管理员 | 团购管理首页 |
| MERCHANT | 商家 | 商品管理首页 |
| ADMIN | 系统管理员 | 系统管理首页 |

### 3.2 登录逻辑

用户在登录页面选择角色后，前端将用户名、密码和角色一起提交给后端。

后端验证：

1. 用户名是否存在；
2. 密码是否正确；
3. 用户账号状态是否正常；
4. 用户角色是否与选择角色一致。

验证成功后，后端返回：

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "token": "jwt-token",
    "userId": 1,
    "username": "zhangsan",
    "role": "USER",
    "homePath": "/user/home"
  }
}
```

前端收到结果后：

1. 保存 token；
2. 保存用户信息；
3. 根据 `role` 或 `homePath` 跳转到对应首页。

### 3.3 不同角色跳转页面

```text
USER     → /user/home
LEADER   → /leader/dashboard
MERCHANT → /merchant/products
ADMIN    → /admin/dashboard
```

## 4.3 前端权限控制设计

前端需要通过路由守卫实现权限控制。

核心逻辑：

```javascript
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  const role = localStorage.getItem('role')

  if (to.path === '/login') {
    next()
    return
  }

  if (!token) {
    next('/login')
    return
  }

  if (to.path.startsWith('/user') && role !== 'USER') {
    next('/login')
    return
  }

  if (to.path.startsWith('/leader') && role !== 'LEADER') {
    next('/login')
    return
  }

  if (to.path.startsWith('/merchant') && role !== 'MERCHANT') {
    next('/login')
    return
  }

  if (to.path.startsWith('/admin') && role !== 'ADMIN') {
    next('/login')
    return
  }

  next()
})
```

## 七、后端核心业务逻辑设计

## 7.1 登录业务逻辑

```text
1. 接收 username、password、role
2. 根据 username 查询 user 表
3. 判断用户是否存在
4. 判断密码是否正确
5. 判断 user.role 是否等于前端选择的 role
6. 判断用户状态是否正常
7. 生成 JWT Token
8. 根据角色返回对应 homePath
```

角色对应首页：

```text
USER     → /user/home
LEADER   → /leader/dashboard
MERCHANT → /merchant/products
ADMIN    → /admin/dashboard
```

---

## 7.2 拼团业务逻辑

### 发起拼团

```text
1. 用户选择商品和拼团活动
2. 后端检查活动是否有效
3. 检查商品库存是否充足
4. 创建 group_instance 记录
5. 创建 orders 记录
6. 当前拼团人数 current_count = 1
7. 返回订单信息
```

### 参与拼团

```text
1. 用户选择已有拼团
2. 后端检查拼团是否仍在有效期
3. 检查拼团是否已满
4. 创建 orders 记录
5. group_instance.current_count + 1
6. 如果 current_count 达到 required_count，则拼团成功
7. 更新该 group_id 下所有订单状态
```

---

## 7.3 免拼业务逻辑

系统不单独建免拼券表，采用简化方式处理：

1. 在 `group_activity` 表中设置 `allow_free_group`。
2. 在 `group_activity` 表中设置 `free_group_limit`。
3. 用户选择免拼购买时，订单类型设为 `FREE_GROUP`。
4. 订单不需要等待拼团人数满员，直接进入支付流程。

免拼订单：

```text
orders.order_type = FREE_GROUP
orders.group_id = null
```

---

## 7.4 订单支付业务逻辑

```text
1. 用户提交订单
2. 系统生成 order_no
3. 系统计算 total_amount
4. 用户点击支付
5. 后端创建 payment 支付记录
6. 支付成功后更新 orders.pay_status
7. 更新 orders.order_status
8. 扣减 product.stock
```

---

## 7.5 商家发货业务逻辑

```text
1. 商家查看已支付订单
2. 商家点击发货
3. 后端更新 orders.delivery_status
4. 后端更新 orders.order_status 为已发货或待自提
5. 系统通知团长和用户
```

---

## 7.6 团长核销业务逻辑

```text
1. 商家发货到自提点
2. 团长查看待取货订单
3. 团长发送取件码
4. 用户到自提点出示取件码
5. 团长输入取件码
6. 后端根据 pickup_code 查询订单
7. 校验订单状态是否为待自提
8. 更新 pickup_status 为已取货
9. 更新 order_status 为已完成
10. 写入通知表 notice
```

---

## 7.7 退款售后业务逻辑

```text
1. 用户选择订单申请退款
2. 后端生成 refund 记录
3. 订单状态更新为退款中
4. 管理员审核退款申请
5. 审核通过则更新 refund_status
6. 更新 orders.order_status 为已退款
7. 发送退款通知
```

## 十四、系统实现重点

### 14.1 统一登录是系统入口

系统不需要为四类角色分别设计登录页，而是使用一个统一登录页，通过用户选择的角色完成不同页面跳转。

### 14.2 后端必须校验角色

前端选择角色只是界面操作，真正的权限判断必须放在后端。

例如：

```text
用户选择 ADMIN 登录，但数据库中该用户 role = USER，则后端必须拒绝登录。
```

### 14.3 前端页面可以逐步美化

初期页面只需要完成基本功能：

1. 能登录；
2. 能跳转；
3. 能查询数据；
4. 能新增、修改、删除；
5. 能完成订单和核销流程。

后期再增加图表、卡片、轮播图、状态标签等美化内容。

### 14.4 取件码建议由系统生成

取件码不建议完全由团长手动填写，建议系统在订单进入“待自提”状态时自动生成，团长只负责发送和核销。

### 14.5 支付可以先做模拟支付

课程设计阶段可以不接入真实微信支付或支付宝支付。  
可以设计一个“立即支付”按钮，点击后模拟支付成功，并写入 `payment` 表。

## 十六、总结

本系统前后端实现应围绕“统一登录、角色分流、业务分层、权限控制”展开。

前端通过一个统一登录页完成所有角色登录，然后根据角色进入不同页面。后端通过 Spring Boot 提供接口服务，并根据 JWT Token 和用户角色控制访问权限。数据库使用 10 张精简核心表，既能覆盖社区团购系统的核心业务，又能避免表结构过于复杂。

最终系统应能够完整支持：

1. 普通用户购物、拼团、免拼、支付和取货；
2. 团长管理自提点、发送取件码和核销订单；
3. 商家发布商品、维护库存和发货；
4. 系统管理员管理用户、商品、拼团、订单和售后。
