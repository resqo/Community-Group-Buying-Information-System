# 社区团购管理信息系统精简版设计说明书

## 一、系统概述

社区团购管理信息系统是一个面向社区居民、商家、社区管理员/团长和系统管理员的综合性管理平台。系统以“拼团购物 + 社区自提”为核心业务模式，主要实现商品浏览、拼团购买、免拼购买、订单支付、商家发货、自提点取货、取件码核销、退款售后和系统管理等功能。

本设计方案采用精简数据库设计，在保证系统核心功能完整的前提下，尽量减少数据表数量，降低系统开发难度，适合课程设计、中小型项目开发和管理信息系统分析与设计作业使用。

最终数据库共设计 **10 张核心数据表**，分别为：

1. `user` 用户表  
2. `category` 商品分类表  
3. `product` 商品表  
4. `group_activity` 拼团活动表  
5. `group_instance` 拼团实例表  
6. `orders` 订单表  
7. `payment` 支付记录表  
8. `pickup_point` 自提点表  
9. `refund` 退款售后表  
10. `notice` 通知表

## 二、系统角色设计

系统主要包括四类角色：普通用户、社区管理员/团长、商家、系统管理员。

### 2.1 普通用户

普通用户进入系统后，看到的是类似购物平台的前台页面，可以完成商品浏览、拼团、免拼、下单、支付、自提和退款等操作。

主要功能包括：

| 功能模块 | 功能说明 |
|---|---|
| 注册登录 | 用户注册、登录、修改密码、维护个人信息 |
| 商品浏览 | 查看商品列表、商品分类、商品详情、商品价格、库存等信息 |
| 拼团功能 | 发起拼团、参与拼团、查看拼团状态 |
| 免拼功能 | 使用免拼资格直接购买支持免拼的商品 |
| 订单管理 | 提交订单、支付订单、取消订单、查看订单状态 |
| 自提管理 | 选择自提点、查看取件码、查看取货状态 |
| 退款售后 | 对订单提交退款申请，查看退款处理进度 |
| 通知查看 | 查看拼团通知、取件通知、退款通知和系统公告 |

---

### 2.2 社区管理员/团长

社区管理员也可以称为团长或自提点管理员。该角色本质上是在普通用户基础上增加社区自提管理职责，主要负责自提点维护、到货确认、取件码发送和订单核销。

主要功能包括：

| 功能模块 | 功能说明 |
|---|---|
| 普通购物功能 | 可像普通用户一样浏览商品、参与拼团、下单购买 |
| 自提点管理 | 维护自提点名称、地址、联系电话、营业时间等信息 |
| 到货管理 | 查看商家配送到本自提点的订单，确认商品到货 |
| 取件码管理 | 查看订单取件码，向用户发送取件通知 |
| 订单核销 | 用户到自提点取货后，输入或核验取件码完成核销 |
| 异常处理 | 记录用户未取货、商品破损、商品缺货等异常情况 |

> 设计建议：取件码可以由系统自动生成并存入订单表，团长负责确认到货后通知用户取货，不需要单独设置一张取件码表。

---

### 2.3 商家

商家主要负责商品发布、商品维护、库存管理、订单查看和发货管理。

主要功能包括：

| 功能模块 | 功能说明 |
|---|---|
| 商品管理 | 新增商品、修改商品、删除商品、上下架商品 |
| 商品信息维护 | 维护商品名称、图片、详情、价格、库存等信息 |
| 拼团商品管理 | 设置商品是否参与拼团，配合平台创建拼团活动 |
| 订单查看 | 查看购买本商家商品的用户订单 |
| 发货管理 | 根据订单信息将商品配送至对应自提点 |
| 销售统计 | 查看商品销量、订单数量、销售金额等数据 |

---

### 2.4 系统管理员

系统管理员负责平台整体运营和系统秩序维护，不直接参与具体商品取货业务。

主要功能包括：

| 功能模块 | 功能说明 |
|---|---|
| 用户管理 | 管理普通用户、团长、商家和管理员账号 |
| 商品审核 | 审核商家发布的商品，处理违规商品 |
| 拼团活动管理 | 创建拼团活动，设置拼团人数、活动时间、拼团价格、免拼规则 |
| 订单管理 | 查看平台全部订单，监控订单状态 |
| 支付管理 | 查看订单支付记录和支付状态 |
| 售后管理 | 审核退款申请，处理售后问题 |
| 通知管理 | 发布系统公告、活动通知、取件提醒等 |
| 数据统计 | 统计订单量、交易额、商品销量、用户增长情况等 |

## 四、核心业务流程设计

### 4.1 拼团购买流程

1. 商家发布商品并维护商品价格、库存和详情信息。
2. 系统管理员审核商品。
3. 系统管理员为商品创建拼团活动，设置拼团人数、拼团价格、活动时间和是否支持免拼。
4. 普通用户浏览商品并查看拼团活动。
5. 用户可以选择发起拼团或参与已有拼团。
6. 系统创建或更新拼团实例。
7. 用户提交订单并完成支付。
8. 系统根据订单更新拼团人数。
9. 当拼团人数达到要求时，系统将拼团状态改为“拼团成功”。
10. 商家根据订单进行备货并发货至自提点。
11. 团长确认商品到达自提点。
12. 系统生成或展示订单取件码。
13. 用户凭取件码到自提点取货。
14. 团长核销订单，订单完成。

---

### 4.2 免拼购买流程

1. 用户选择支持免拼的商品。
2. 系统判断该商品是否允许免拼。
3. 系统判断用户是否仍有免拼资格或免拼次数。
4. 用户直接提交免拼订单。
5. 用户完成支付。
6. 商家发货至用户选择的自提点。
7. 团长确认到货并通知用户取货。
8. 用户凭取件码取货。
9. 团长核销订单，订单完成。

---

### 4.3 自提核销流程

1. 用户下单时选择自提点。
2. 商家根据订单将商品配送到指定自提点。
3. 团长确认商品到货。
4. 系统向用户发送取件通知。
5. 用户在个人中心查看取件码。
6. 用户到自提点出示取件码。
7. 团长核验取件码。
8. 系统将订单状态改为“已完成”。

---

### 4.4 退款售后流程

1. 用户在订单页面提交退款申请。
2. 系统生成退款售后记录。
3. 系统管理员查看退款申请。
4. 管理员审核退款原因、订单状态和支付状态。
5. 审核通过后，系统更新退款状态和订单状态。
6. 用户查看退款处理结果。

---

## 五、数据库总体设计

本系统采用精简数据库设计，共设计 10 张核心表。通过字段合并和业务查询方式减少数据表数量。

### 5.1 简化设计原则

| 简化点 | 设计方式 |
|---|---|
| 不单独设计角色表 | 在 `user` 表中使用 `role` 字段区分角色 |
| 不单独设计商家表 | 商家也属于用户，在 `user` 表中用商家相关字段保存店铺信息 |
| 不单独设计社区表 | 社区名称直接保存在用户表和自提点表中 |
| 不单独设计商品图片表 | 商品主图和详情图直接保存在 `product` 表中 |
| 不单独设计订单明细表 | 一个订单默认对应一个商品，商品快照直接保存在 `orders` 表中 |
| 不单独设计拼团成员表 | 通过 `orders.group_id` 查询某个拼团下的所有成员 |
| 不单独设计取件码表 | 取件码直接保存在 `orders.pickup_code` 字段中 |
| 不单独设计免拼券表 | 使用拼团活动表中的免拼字段和订单类型字段完成免拼业务 |

## 6.1 user 用户表

### 表说明

`user` 表用于统一存储普通用户、社区管理员/团长、商家和系统管理员信息。

| 字段名 | 类型 | 说明 |

| status | tinyint | 账号状态：正常 / 禁用 |
| create_time | datetime | 创建时间 |

### 角色说明

| role 值 | 含义 |
|---|---|
| USER | 普通用户 |
| LEADER | 社区管理员/团长 |
| MERCHANT | 商家 |

## 十、设计方案总结

本社区团购管理信息系统采用 10 张核心表的精简数据库设计方案。该方案通过字段合并和业务查询方式减少不必要的数据表，同时仍然能够支持普通用户购物、拼团、免拼、订单支付、商家发货、社区自提、团长核销、退款售后和系统后台管理等核心业务。

最终设计的 10 张表为：

| 序号 | 表名 | 中文名称 | 主要作用 |
|---|---|---|---|
| 1 | user | 用户表 | 存储普通用户、团长、商家、管理员 |
| 2 | category | 商品分类表 | 存储商品分类 |
| 3 | product | 商品表 | 存储商品信息、价格、库存、图片 |
| 4 | group_activity | 拼团活动表 | 存储拼团规则 |
| 5 | group_instance | 拼团实例表 | 存储某一次具体拼团 |
| 6 | orders | 订单表 | 存储订单、商品快照、取件码、核销状态 |
| 7 | payment | 支付记录表 | 存储订单支付记录 |
| 8 | pickup_point | 自提点表 | 存储社区自提点信息 |
| 9 | refund | 退款售后表 | 存储退款申请和处理结果 |
| 10 | notice | 通知表 | 存储系统通知、取件通知、退款通知 |

该方案的主要优点包括：

1. 数据表数量少，结构清晰。
2. 能够覆盖社区团购系统的主要业务功能。
3. 普通用户、团长、商家、系统管理员四类角色职责明确。
4. 订单表集成商品快照、拼团信息、取件码和核销状态，便于课程设计实现。
5. 后续如果系统规模扩大，可以继续拆分角色表、商家表、订单明细表、商品图片表、拼团成员表和取件码表。

因此，本系统最终推荐采用 **10 张核心表** 的数据库设计方案。
